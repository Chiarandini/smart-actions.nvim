1. Overview					*smart-actions-overview*

smart-actions.nvim augments (not replaces) the stock LSP code-action flow
on `gra` with AI-suggested actions on `grA`. Scope is configurable, actions
are grouped into pluggable categories, and the provider layer resolves
to the local Claude Code CLI first and the Anthropic Messages API as
fallback.

v1 ships the `quickfix` category only.

==============================================================================
2. Setup					*smart-actions-setup*

>lua
    require("smart_actions").setup({
      default_scope = "ask",        -- or "auto" | "line" | "function" | ...
      provider = nil,               -- nil = auto-detect
      include_diagnostics = true,
      keymap = "grA",
    })
<

3. Keymaps					*smart-actions-keymaps*

`grA` (normal, visual)		Run a smart action with the configured scope.
				If `default_scope = "ask"`, the scope picker
				is shown first.

Inside the results picker: ~
  <CR>		apply the selected action(s). If nothing is Tab-marked,
		applies the hovered one. If one or more actions were
		marked with <Tab>, applies them all sequentially with a
		single undo unit; any hunk that can't land on the mutated
		buffer is SKIPPED and reported (N of M applied, K skipped).
  <Tab>		toggle-select the hovered action for multi-apply.
		<S-Tab> toggles and moves up.
  <C-e>		open the diff in a scratch buffer for hand-editing;
		 `:w` applies the edited patch, `:q!` cancels. Always
		targets the HOVERED action, not the multi-select set —
		editing a bundle of diffs doesn't make sense.
  <Esc>		cancel without applying

Inside the edit-scratch buffer: ~
  :w		apply the (possibly-edited) patch
  :q!		cancel

Inside the explain float: ~
  a / <CR>	close the float and run the default action category
		(quickfix) on the same scope — i.e. "OK, now fix it"
  q / <Esc>	close the float without further action; cancels the
		eager background action if one is running
All live keymaps are also shown in the float's bottom border.

						*smart-actions-eager*
Eager action after explain (opt-in): ~

When `eager_action_after_explain = true` in setup(), the default action
category (e.g. quickfix) starts streaming in the background the moment
an explain stream finishes. The explain float stays open for reading.
If the user then presses `a`/<CR>, the picker opens immediately (or as
soon as the background stream completes). If the user presses `q`/<Esc>,
the in-flight background action is cancelled.

Trade-off: token cost roughly doubles for any explain invocation the
user dismisses before the background action finishes. Leave disabled
(the default) unless you frequently chain explain → fix and want to
hide the latency inside your reading time.

4. Commands					*smart-actions-commands*

:SmartAction [scope]		Run a smart action, optionally forcing scope.
:SmartActionCancel		Cancel the in-flight request.
:SmartActionLastDiff		Show the last applied action's full diff text
				in a vertical split. Useful when the picker's
				inline preview (delta-rendered) hides +/-
				markers and you want to confirm what the AI
				actually produced.
:SmartActionExplain [scope]	Explain the code/diagnostic under cursor
				(prose, streamed into a floating window; no
				diff, no apply). Complements quickfix: tells
				you WHY something is off instead of fixing it.
:SmartActionSuppress [scope]	Produce suppression-comment actions for LSP
				diagnostics (language-appropriate:
				`# pyright: ignore[...]`, `// @ts-expect-error`,
				`#[allow(...)]`, etc.). Does not modify logic.
				Returns nothing when no diagnostics exist.
:SmartActionRefactor [scope]	Propose behaviour-preserving refactors
				(extract, inline, simplify, replace-mutation-
				with-functional). Returns nothing when there
				is no clear refactor opportunity.
:SmartActionTests		Generate ONE test for the function closest
				to the cursor, appended to the current file.
				Scope is forced to "file" so the AI sees the
				full context. Framework auto-detected from
				file extension + existing imports (pytest,
				vitest, #[test], Go testing, plenary).
				Multi-file test-file placement is a deferred
				v2.x feature; for now the test is appended
				to the current file and can be hand-moved.
:SmartActionReview [scope]	Broad code review — returns a mix of
				blockers, suggestions, nits, and questions.
				Items may be fixes (have a diff) or
				observations (rationale-only). Observations
				surface in the preview pane as markdown and
				don't apply when confirmed; they surface
				their rationale as a notification instead.

5. Scopes					*smart-actions-scopes*

line | function | file | folder | project | auto | visual | ask

Each concrete scope is also exposed as a callable so users can bind their
own keys:
>lua
    vim.keymap.set("n", "<leader>Saf", function()
      require("smart_actions").run_scope("function")
    end, { desc = "smart action ([f]unction scope)" })
<
					*smart-actions-payload-caps*
All scopes are bounded:
  max_payload_chars    = 6000    -- hard cap on scope.text
  max_files            = 2000    -- cap on folder/project enumeration
  max_file_bytes       = 200000  -- per-file read cap (~200 KB)
  file_scan_timeout_ms = 500     -- max time rg may take to enumerate files

When a cap fires:
  - single-buffer scopes (line/function/file/visual) set scope.truncated = true
    and append a "-- + N chars truncated (budget M) --" marker
  - folder/project set scope.files_omitted > 0 (or -1 when the rg stream was
    cut short by max_files or file_scan_timeout_ms, exact overflow unknown)
    and prepend a "-- + N file(s) beyond max_files --" notice
  - the files[] list always reflects what WAS enumerated; truncated individual
    files carry .truncated = true

Folder/project enumeration uses `rg --files` streamed through vim.system
and is killed the instant it has `max_files` paths or `file_scan_timeout_ms`
has elapsed — whichever comes first — so nvim never freezes on huge trees.

						*smart-actions-diagnostics*
Diagnostics attached to a scope: ~

Every Scope object carries four diagnostic fields:

  scope.trigger.diagnostics          -- all diagnostics on the cursor LINE
                                        (back-compat; superset of the two
                                        partitions below)
  scope.trigger.diagnostics_at_col   -- subset whose range contains the
                                        cursor COLUMN (the focal set — the
                                        thing the user is LOOKING at)
  scope.trigger.diagnostics_on_line  -- `diagnostics` minus `diagnostics_at_col`
  scope.diagnostics                  -- vim.diagnostic.get(bufnr) filtered to
                                        entries whose range overlaps
                                        scope.start .. scope.end_ (single-
                                        buffer scopes); for folder/project,
                                        additionally walks loaded buffers
                                        whose path is in scope.files

Unloaded files yield no diagnostics — LSPs only analyze open buffers.

The quickfix category surfaces all three trigger tiers in priority order:
>
    Diagnostic AT cursor column (highest priority):
      line N col M: [severity] message (source)

    Other diagnostics on cursor line:
      line N col M: [severity] message (source)

    Other diagnostics in this scope:
      line K col M: [severity] message (source)
<
The AI is instructed to target the highest-priority non-empty tier first.
Deduplication keyed on (lnum, col, message, source) ensures nothing repeats.

6. Categories					*smart-actions-categories*

Shipped: `quickfix`, `explain`, `suppress`, `refactor`, `tests`, `review`.
Additional categories register via the same interface.

						*smart-actions-quickfix-modes*
Quickfix has two prompt modes, picked automatically from scope shape: ~

  Cursor-focus (default) — any non-visual scope. The AI biases toward the
  trigger line using the three-tier diagnostic layout (see
  |smart-actions-diagnostics|). Cap: 3 actions. Use when you're pointing
  at something and want it fixed; include `function` / `file` / `folder`
  scope to widen the CONTEXT the AI sees, not the fix surface.

  Region-fix — triggered when `scope.label == "visual"` (i.e. you pressed
  grA over a visual selection). No cursor-proximity bias; the selection
  IS the "here, fix all of this" gesture. Cap scales with the diagnostic
  count in the selection:
>
    cap = clamp(3, n_diagnostics + 2, quickfix_region_max_actions)
<
  So a clean selection still gets 3, 5 diagnostics ⇒ 7, 20 diagnostics ⇒
  10 (default ceiling). Raise `quickfix_region_max_actions` if you
  regularly select large regions with many lint warnings.

7. Providers					*smart-actions-providers*

Built-in:
  claude_code	uses the local `claude` CLI (reuses your Claude Code auth)
  anthropic	direct Messages API via curl + streaming SSE
  openai	generic OpenAI-compatible /chat/completions via curl + SSE
		— covers OpenAI, Ollama, LM Studio, OpenRouter, Groq, Gemini
		(compat endpoint), xAI, Together, and any other service
		speaking the OpenAI chat-completions wire format

API key resolution (anthropic):
  1. `lua/secrets.lua`  -> `{ anthropic = { api_key = "..." } }`
  2. `$ANTHROPIC_API_KEY`

API key resolution (openai, per-spec):
  1. provider_config.<id>.api_key (inline, e.g. from lua/secrets.lua)
  2. os.getenv(provider_config.<id>.api_key_env)
     — default env var is "OPENAI_API_KEY"
  3. If api_key_env = "" AND api_key is unset, auth is skipped (for local
     endpoints like Ollama / LM Studio / llama.cpp that don't gate on a
     bearer token).

						*smart-actions-openai-recipes*
Using `openai` with other services: ~

Set `provider = "openai"` and override `provider_config.openai`:

  OpenAI		base_url = "https://api.openai.com/v1"
			api_key_env = "OPENAI_API_KEY" (default)
  Ollama		base_url = "http://localhost:11434/v1"
			api_key_env = "" (no auth)
  LM Studio		base_url = "http://localhost:1234/v1"
			api_key_env = ""
  OpenRouter		base_url = "https://openrouter.ai/api/v1"
			api_key_env = "OPENROUTER_API_KEY"
  Groq			base_url = "https://api.groq.com/openai/v1"
			api_key_env = "GROQ_API_KEY"
  Gemini (compat)	base_url = "https://generativelanguage.googleapis.com/v1beta/openai"
			api_key_env = "GEMINI_API_KEY"
  xAI (Grok)		base_url = "https://api.x.ai/v1"
			api_key_env = "XAI_API_KEY"

See README.md "Using other AI models" for full copy-paste recipes.

						*smart-actions-openai-define*
Multiple OpenAI-compatible endpoints simultaneously: ~

If you need more than one endpoint live at the same time (e.g. OpenAI
for cloud + Ollama for offline fallback in `probe_order`), use the
factory exposed on the openai provider module:
>lua
    local openai = require("smart_actions.providers.openai")
    require("smart_actions.providers").register(
      openai.define({
        id          = "ollama",
        base_url    = "http://localhost:11434/v1",
        model       = "qwen3-coder",
        api_key_env = "",
      })
    )
<
Each `define(spec)` returns a first-class provider with its own `id`,
slotting into `probe_order` and `provider_config[id]` like any built-in.

Pluggable: third-party plugins can register arbitrary providers — use
the factory for OpenAI-compatible services, and |smart-actions-register-provider|
for anything else.

						*smart-actions-register-provider*
>lua
    require("smart_actions.providers").register({
      id = "my_cli",
      display_name = "My custom LLM CLI",
      probe  = function() return vim.fn.executable("my-llm") == 1 end,
      stream = function(req, cb) ... return cancel_fn end,
    })
<

`req.opts` is populated from `config.provider_config[id]` so each provider
reads its own slot without polluting the top-level config.

8. Context					*smart-actions-context*

Context providers inject *project knowledge* — rules, idioms, house
conventions — into the system prompt ahead of every `grA` request. They're
separate from scope (the code itself) and categories (the task shape).

Same registry pattern as providers/categories: five built-ins auto-load at
setup(); users register their own via register().

						*smart-actions-context-builtins*
Built-ins: ~

  id                 priority   triggers on
  ─────────────────  ────────   ─────────────────────────────────────────
  claude_md          90         any CLAUDE.md up the ancestor chain
  agents_md          85         any AGENTS.md up the ancestor chain
  cursorrules        80         any .cursorrules up the ancestor chain
  neovim_plugin      70         plugin/*.lua | lua/<name>/init.lua | *.nvim
  language_default   10         filetype is one of the known ones

						*smart-actions-context-shape*
Provider shape: ~

A context provider is a table:
>lua
    {
      id       = "my_rules",
      priority = 100,                       -- higher wins on truncation
      detect   = function(root) -> boolean, -- does this apply here?
      gather   = function(scope) -> string, -- the block to inject
    }
<
- `detect(root)` is cheap; called once per `grA`. Return false to skip.
- `gather(scope)` returns the text block; return "" to skip (useful when
  `detect` is coarse but `gather` is filetype-keyed).
- `priority` defaults to 0; higher blocks are kept first when budget is
  tight; lower blocks are dropped.

						*smart-actions-context-register*
Registering one: ~

Drop this in your user-config `config = function(_, opts) ... end`:
>lua
    require("smart_actions.context").register({
      id = "my_project", priority = 200,
      detect = function(root)
        return vim.uv.fs_stat(root .. "/.myproject") ~= nil
      end,
      gather = function(scope)
        return "House rules: tabs not spaces, no emojis, prefer explicit returns."
      end,
    })
<

						*smart-actions-context-budget*
Config knobs: ~

>lua
    context = {
      enabled   = true,
      max_chars = 4000,          -- total budget; lower-priority dropped first
      allowlist = nil,           -- nil = all matching providers
                                 -- or { "claude_md", "my_project" }
      denylist  = nil,           -- e.g. { "language_default" } to mute one
    }
<
When the budget is exceeded, the assembled string ends with:
  `<!-- N context block(s) omitted (budget M chars) -->`

						*smart-actions-context-native*
Natively-handled context: ~

Some AI providers already consume certain context sources on their own. The
built-in `claude_code` provider declares:
>lua
    M.handles_natively = { "claude_md", "agents_md" }
<
which tells the context assembler to SKIP those sources when `claude_code`
is the active provider (it auto-discovers them from the filesystem). If
you're building a new provider that already handles e.g. `cursorrules`,
add its id to your `handles_natively` list.

						*smart-actions-context-inspect*
Inspecting what the AI sees: ~

>lua
    :lua =require("smart_actions.context").assemble(
           require("smart_actions.scope").get("function"),
           require("smart_actions.providers").active())
<
Prints the exact `<project_context>` block that'll ride with the next
request.

9. Known limitations				*smart-actions-limits*

What v1 deliberately does NOT do (planned for v2):

  Multi-file edits ~
    The applier targets the cursor's buffer only. folder/project scopes
    widen what the AI SEES, but diffs that edit other files are silently
    ignored. If you need a cross-file refactor, pick one file at a time.

  Streaming picker entries ~
    Actions are accumulated during the stream, then shown when the AI
    finishes. Partial streaming into an already-open picker isn't wired
    yet (Snacks.picker doesn't expose an append API as of this version).

  Async file enumeration ~
    folder/project scope's `rg --files` streams through vim.system and
    kills rg the instant it has `max_files` paths or `file_scan_timeout_ms`
    has elapsed. The main-thread wait is bounded by the timeout so nvim
    never freezes on huge trees — see |smart-actions-payload-caps|.

  Additional AI providers ~
    Only claude_code CLI and anthropic API ship. OpenAI / Ollama / Gemini
    would each be one file under lua/smart_actions/providers/ — see
    |smart-actions-register-provider|.

  Action severity tiering ~
    Every action shows at equal weight. A v2 "core" vs "optional" tier is
    sketched in the plan but not implemented.

  Request timeout guard ~
    Configurable via config.provider_timeout_ms (default 120000 = 2 min).
    Set to 0 to disable. Fires on_error("… timed out after Nms") and
    clears the busy spinner.
